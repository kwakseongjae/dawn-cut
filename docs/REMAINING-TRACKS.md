## A. 한국어 뉴럴 TTS (감정형 음색)

### 현황
- 기본 엔진 = macOS **`say`** (오프라인·내장). 한글 입력 시 한국어 보이스(유나/에디 등)로 자동 전환.
  속도·톤·스타일(차분/보통/활기참) 조절 가능. **단, `say`는 진짜 감정(분노/기쁨/속삭임) 음색은 불가** —
  피치·속도 변조로 '느낌' 근사까지만.
- Piper(뉴럴) 경로는 **인프라만 깔려 있음**: `isPiperAvailable()` 감지, 엔진 배지("뉴럴(Piper)"),
  `pnpm setup:tts-neural`(영어 모델 설치). 영어는 실제 뉴럴 합성 동작.
- **한국어 뉴럴은 막혀 있음** — 조사 결과:
  1. 공식 `rhasspy/piper-voices`에 **한국어(ko) 보이스가 아예 없음**(2026 기준, 44개 언어 중 ko 부재).
  2. 유일한 커뮤니티 ko 모델(`neurlang/kss`)은 표준 Piper가 아니라 **Rust(pygoruut) 런타임**을 요구 →
     현재 `synthesizeTts`의 `piper --model/--output_file/stdin` 인터페이스로 실행 불가.
  3. 그 모델 라이선스가 **CC-BY-NC(비상업)** → 오픈소스 에디터 기본 동봉/권장에 부적합.

### 옵션
| 안 | 방법 | 장점 | 단점 / 결정 필요 |
|---|---|---|---|
| **A1** | `piper-rs`(pygoruut) 별도 통합 + `neurlang/kss` | 진짜 한국어 뉴럴, 오프라인 | Rust 바이너리 빌드·배포, exec 인터페이스 전면 변경, **CC-BY-NC(상업 불가)** |
| **A2** | 다른 오프라인 뉴럴(Coqui XTTS / MeloTTS 등) | 한국어 지원, 감정/클로닝 | Python+torch 의존(수백 MB~GB), 배포 용량·기동시간 큼 |
| **A3** | 클라우드 TTS(ElevenLabs/Typecast 등) | 최고 품질·감정 | **오프라인·프라이버시 해자 정면 위배** → opt-in 한정 |
| **A4** | `say` 유지 + 톤/속도 프리셋 확대 | 0 의존성, 즉시 | 진짜 감정 음색 불가(현 한계) |

### 권고
- **단기: A4 유지** — 한국어는 `say`(유나) + 스타일 프리셋이 현실적 최선. 추가 비용 0.
- **중기: A1을 "실험적 옵트인"으로** — 상업적으로 안전한 한국어 Piper 모델이 나오거나 라이선스를 확보하면
  착수. 그 전까지 `setup:tts-neural`은 영어 전용 + 한국어 한계 명시(현재 상태).
- **클라우드(A3)는 기본 비채택** — 정 필요하면 명시적 opt-in + 출처 고지로만.

### 결정해야 할 것
1. **비상업(CC-BY-NC) 모델을 수용**할 것인가? (오픈소스 배포·상업화 계획과 충돌 가능)
2. **Rust/추가 바이너리 배포**(piper-rs)를 감수할 것인가? (빌드·서명·용량)
3. 한국어 뉴럴의 **목표 품질·용량 상한**(예: ≤100MB 모델)

### 착수 시 작업 분해(A1 기준, 대략)
- `scripts/setup-tts-neural.sh`에 piper-rs 빌드 + ko 모델(동의 기반) 분기 — `중`
- `sidecar/tts`: piper-rs CLI 인터페이스 어댑터(stdin/인자 다름) + 한국어 분기 — `중상`
- 라이선스/고지 UI(비상업 경고) — `소`
- 검증: ko 합성 → whisper 역전사 recall, 길이/감정 파라미터 — `중`

## B. 클라우드 GIF 검색 (GIPHY / Tenor)

### 현황
- '스티커·GIF' 패널 = (1) 이모지 스티커, (2) **로컬 생성 모션 스티커 12종**(스피너·🔥·❤️·👍·구독 등,
  `pnpm assets:gif`로 결정적 생성·번들), (3) **내 GIF/이미지 드롭존**(커스텀 업로드 — 실제 애니 GIF 합성 동작).
- **클라우드 GIF 검색은 의도적으로 없음** — 오프라인·프라이버시 포지셔닝(영상이 기기를 안 떠남)과 충돌.

### 옵션
| 안 | 방법 | 장점 | 단점 / 결정 필요 |
|---|---|---|---|
| **B1** | 현행 유지(로컬 팩 + 커스텀 업로드) | 프라이버시·오프라인 해자, 0 비용 | 검색 가능한 방대한 라이브러리는 없음 |
| **B2** | **로컬 팩 대폭 확장**(20~50종) + 태그/검색 UI | 클라우드 없이 풍부함, 결정적 | 콘텐츠 제작·큐레이션 노동 |
| **B3** | GIPHY/Tenor **opt-in 토글**(기본 off) | 무한 라이브러리 | 네트워크 송신(검색어 유출), API 키·약관·요금, 해자 약화 |

### 권고
- **B1 + B2** — 클라우드 없이 로컬 팩을 늘리고(같은 `scripts/make-gif-stickers.ts` 루프) 20종↑이면
  태그/검색을 붙인다. 프라이버시 메시지를 지키면서 충분히 풍부.
- **B3는 명시적 opt-in으로만** — '온라인 GIF(외부 서비스 사용)' 토글을 끄는 게 기본값. 켤 때
  "검색어가 외부로 전송됩니다" 고지 + 출처 표기. 제품 포지셔닝상 권하지 않음.

### 결정해야 할 것
1. **프라이버시 포지셔닝 vs 콘텐츠 다양성** 중 무엇을 우선하는가?
2. (B3 시) GIPHY/Tenor **API 약관·요금·키 배포** 방식 — 사용자 키 입력 vs 동봉 키(유출 위험)
3. 로컬 팩(B2)을 몇 종까지, 어떤 카테고리(리액션/화살표/구독유도/이모지모션)로 큐레이션할지

### 착수 시 작업 분해
- **B2**: `scripts/make-gif-stickers.ts`에 drawer 추가(루프 확장) + 20종↑이면 패널에 태그/검색 — `소~중`
- **B3**: `online:gifSearch` IPC(opt-in 가드) + 검색 UI + 캐시 + 고지/설정 토글 — `중상`
